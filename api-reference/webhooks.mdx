<Note>
  **Available today:** `session.analysis_complete`. Other events (`session.ended`,
  `session.transcript_ready`, `recording.ready`) are not yet delivered — use
  [polling](/api-reference/session-lifecycle) or [email notifications](/capabilities/notifications) for those.
</Note>

## Setup

Configure a webhook URL for your scenario via the **Settings** tab or the session
options API:

```bash
curl -X PUT https://api.waterr.ai/v1/scenarios/{scenarioId}/session-options \
  -H "Authorization: Bearer wai_<your_key>" \
  -H "Content-Type: application/json" \
  -d '{
    "webhook_url": "https://your-app.com/webhooks/waterr"
  }'
```

Once set, every analysis that completes for a meeting on that scenario will fire
a `session.analysis_complete` POST to your URL.

## Event payload

Every webhook POSTs a JSON body with this envelope:

```json
{
  "id": "9b6c…",
  "event": "session.analysis_complete",
  "created_at": "2026-05-31T09:25:00.000Z",
  "data": {
    "meeting_id": "meeting-uuid",
    "scenario_id": "scenario-uuid",
    "analysis": {
      "analysis_id": "analysis-uuid",
      "average_score": 82,
      "total_score": 412,
      "summary": "The candidate demonstrated strong …",
      "call_summary": "30-minute system design conversation …",
      "strengths": "…",
      "growth_areas": "…",
      "recommendations": "…",
      "goal_results": [
        { "goal": "Problem Solving", "result": { "score": 4, "feedback": "…" } }
      ],
      "highlights": [
        { "title": "Identified the bottleneck early", "timestamp": 412 }
      ],
      "conversation_length": 187
    }
  }
}
```

The `data.analysis` object mirrors what `GET /analyses/meeting/{meetingId}` returns,
so any code that already consumes the analysis API works unchanged.

## Verifying the signature

Every request includes a `X-Waterr-Signature` header containing an HMAC-SHA256
of the raw request body, signed with your webhook secret.

```
X-Waterr-Signature: sha256=4c1f8a…
```

Verify it before trusting the payload (Node.js):

```js
const crypto = require('crypto');

app.post('/webhooks/waterr', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-waterr-signature'];
  const expected =
    'sha256=' +
    crypto
      .createHmac('sha256', process.env.WATERR_WEBHOOK_SECRET)
      .update(req.body)
      .digest('hex');

  if (
    !signature ||
    !crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))
  ) {
    return res.status(401).send('invalid signature');
  }

  const event = JSON.parse(req.body);
  // ... handle event.event === 'session.analysis_complete'
  res.sendStatus(200);
});
```

<Warning>
  Verify using the **raw** request body, not a re-stringified JSON object. Body
  parsers like `express.json()` will mutate whitespace and break the HMAC.
</Warning>

## Delivery behavior

- Fired once per completed analysis, fire-and-forget.
- Timeout: **5 seconds**. Responses are not parsed — any 2xx is success.
- **No retries** in the current MVP. If your endpoint is down when the analysis
  completes, you'll miss that event. Until retries ship, treat the webhook as a
  *fast path* and keep polling `GET /analyses/meeting/{meetingId}` as a backstop
  for high-stakes flows.
- Failures are logged on our side with the response status (or error) for debugging.

## Events

| Event | Status | Fires when |
|-------|--------|------------|
| `session.analysis_complete` | **Available** | AI analysis with goal scores is saved |
| `session.ended`             | Planned       | Meeting ends (before analysis runs) |
| `session.transcript_ready`  | Planned       | Full transcript is finalized |
| `recording.ready`           | Planned       | Video recording is uploaded |
